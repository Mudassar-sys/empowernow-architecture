Phase 1: Harden Current Stack

Weeks 1-2 | Fixed Price: $3,500 | Security baseline before migration

Credential Rotation

Rotate all service credentials, database passwords, API keys. Implement automated rotation schedule. Eliminate hardcoded secrets from compose files and environment variables.

Close Exposed Data Ports

Audit all exposed ports in Docker Compose. Remove direct database port exposure. Implement network-level isolation between services. Only Traefik receives external traffic.

Fail-Closed Health Checks

Replace permissive health checks with fail-closed implementations. Services must prove readiness by verifying downstream dependencies. Unhealthy services get zero traffic.

Split Valkey Planes

Separate into two distinct instances: ephemeral cache plane (eviction-safe) and coordination plane (persistent, for leader election and distributed locks). Different persistence configs.

Image Digest Pinning

Pin all container images to SHA256 digests. No :latest or mutable tags. Create a manifest of current images with verified checksums. Foundation for supply-chain security.

Backup/Restore Testing

Implement and test backup/restore for PostgreSQL (pg_dump + WAL), Neo4j (neo4j-admin dump), ClickHouse (backup + S3), OpenBao (Raft snapshots). Verify restore into clean environment.

Distributed Systems Design: Fencing & Exactly-Once

We understand why "exactly-once" is a claim you refuse to make

SINGLETON WORKER FENCING DESIGN

The problem: A singleton worker writes to a database and must never run twice concurrently. Network partitions, GC pauses, and DB failovers can all create split-brain conditions.

Our approach:

  1. Leader election via Valkey coordination plane with TTL-bounded leases
  2. Fencing tokens: monotonically increasing token issued with each lease grant
  3. Database writes include fencing token in a conditional update (WHERE fence_token >= $current)
  4. Stale leader's writes are rejected by the database even if the process doesn't know it lost the lease

Failure modes we handle:

  • Network partition: Lease expires, standby promotes, old leader's writes are fenced out
  • GC pause: Same as partition. Process wakes after TTL, its token is stale
  • DB failover: New primary inherits fencing token state via replication. No window for dual writes

TRANSACTIONAL OUTBOX PATTERN

Why not publish directly to Redpanda? Because a service that writes to PostgreSQL AND publishes to Redpanda has a dual-write problem. If the DB commit succeeds but the publish fails, you lose the event. If the publish succeeds but the commit fails, you have a phantom event.

The outbox approach:

  1. Service writes business data + outbox row in a single DB transaction
  2. Dedicated publisher reads outbox, publishes to Redpanda, marks row as sent
  3. Consumers are idempotent: dedup key in every message, consumer tracks processed keys
  4. Result: at-least-once delivery with idempotent consumers = effectively-once processing

We never claim exactly-once. We design for at-least-once delivery with idempotent, reentrant consumers. The outbox + idempotent consumer pattern gives you the same end result without the impossible guarantee.

Drain-and-redrive: During migration, we replay from outbox tables (not Redpanda retention) to guarantee zero message loss during cutover.
